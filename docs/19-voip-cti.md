# VoIP / CTI — شناسایی خودکار تماس ورودی (Screen Pop)

## وضعیت فعلی (ثبت‌شده 2026/07/19)
- خط تلفن: سرویس آسیاتل (آسیاتک) بر بستر SIP — پشتیبانی فقط خط را رجیستر و تحویل می‌دهد
- پاسخگویی: IP Phone سخت‌افزاری
- یک سرور/دستگاه جداگانه به‌صورت فیزیکی کنار تلفن‌ها با «محیط مدیریت» که داخلی‌ها و
  تعداد تماس همزمان در آن تنظیم می‌شود ← به احتمال زیاد این خودش PBX است
- حساسیت: تماس‌ها منبع اصلی درآمد مجموعه‌اند؛ هر تغییری فقط در روز تعطیل و با برنامه‌ی بازگشت

## هدف فاز ۱
هنگام تماس ورودی، شماره‌ی تماس‌گیرنده به‌صورت خودکار به CRM برسد و برای پذیرش
کارت «تماس ورودی» نمایش داده شود: مشتری شناخته‌شده (با سوابق) یا شماره جدید.
تماس‌هایی که مشتری نیستند در کارت مکالمه/سوال ثبت می‌شوند؛ اگر مشتری بود دکمه «پذیرش»
مستقیم به فرم قبض پذیرش با اطلاعات پیش‌پرشده می‌رود.

## معماری هدف
تماس ورودی → PBX (دستگاه موجود) → رویداد تماس (CDR/Webhook/AMI/API)
→ سرویس پل کوچک → POST به CRM (endpoint دریافت رویداد تماس)
→ نمایش Real-time کارت تماس برای پذیرش

## اطلاعاتی که باید جمع شود (چک‌لیست برای همام)
### الف) شناسایی دستگاه مدیریت تلفن‌ها (مهم‌ترین)
- [ ] برند و مدل دستگاه: روی برچسب خود دستگاه، یا عنوان/لوگوی صفحه‌ی ورود محیط مدیریت
      (نمونه‌های رایج در ایران: Grandstream UCM، Yeastar S/P، Newrock، Simotel، Issabel/FreePBX)
- [ ] آدرس IP محیط مدیریت + دسترسی ادمین
- [ ] نسخه Firmware (معمولاً در منوی About / System Information / Status)
- [ ] اسکرین‌شات از منوهای اصلی محیط مدیریت
### ب) قابلیت‌های اتصال (در منوهای دستگاه جستجو کن)
- [ ] وجود هر یک از این کلمات در تنظیمات: API, AMI, CDR, Webhook, Event, CTI, HTTP Callback
- [ ] بخش گزارش تماس‌ها (Call Log / CDR) — آیا خروجی/دانلود دارد؟
### ج) اطلاعات خط آسیاتل (از پنل آسیاتک یا برگه تحویل خط؛ در صورت نبود از پشتیبانی بگیر)
- [ ] آدرس سرور SIP (domain/IP) و پورت
- [ ] نام کاربری (معمولاً همان شماره) و رمز رجیستر خط
### د) شبکه
- [ ] IP دستگاه PBX در شبکه داخلی و اینکه آیا به اینترنت (سرور CRM) دسترسی دارد
- [ ] تعداد داخلی‌ها و اینکه IP Phone به این دستگاه رجیستر است یا مستقیم به آسیاتل

## اصول اجرای امن (توافق‌شده)
1. هیچ تغییری روی مسیر تماس تا اطلاع ثانوی — فقط جمع‌آوری اطلاعات
2. فاز اول اتصال «فقط‌خواندنی» است (خواندن CDR/رویداد) → ریسک صفر برای تماس‌ها
3. هر تغییر پیکربندی احتمالی: فقط روز تعطیل + بکاپ کانفیگ PBX قبل از تغییر + برنامه بازگشت
4. تست با یک شماره قبل از اتصال کامل

## فازها
1. شناسایی دستگاه (چک‌لیست بالا) → تعیین روش اتصال
2. Endpoint دریافت رویداد تماس در CRM + کارت Real-time «تماس ورودی» برای پذیرش
3. سرویس پل روی همان سرور PBX یا سرور CRM (بسته به قابلیت دستگاه)
4. اتصال کارت تماس به: سابقه مشتری / ثبت مکالمه‌ی غیرمشتری / دکمه پذیرش با پیش‌پر کردن
